# Аудит продукта: Chef OS

## 1. Executive summary (Общее резюме)
Chef OS — это многообещающий концепт с сильным фокусом на реальные операционные процессы кухни. Продукт правильно понимает, что шеф-поварам нужен не «еще один сложный ERP», а быстрый мобильный инструмент для управления сменой (Shift Command). Однако на данный момент это скорее кликабельный прототип, чем готовый SaaS.
Главная проблема текущей версии: гигантский монолитный frontend (`src/main.jsx` на ~1800 строк), отсутствие полноценной логики расчета себестоимости (Food Cost) и редактирования ТТК (технико-технологических карт), а также отсутствие ролевой модели в интерфейсе. Продукт выглядит хорошо, но ему не хватает глубины в части работы с ингредиентами и ценообразованием, чтобы за него начали платить владельцы бизнеса.

## 2. Оценки продукта (от 1 до 10)
- **UX:** 7/10 (Хорошие задумки, но перегруженность стейтами в монолите может привести к багам при масштабировании).
- **Mobile usability:** 8/10 (Кнопки крупные, навигация внизу, карточки удобные).
- **Chef usefulness:** 6/10 (Полезно для смены, но нет инструментов для планирования меню и калькуляции).
- **Business value:** 4/10 (Пока нет калькуляции фудкоста и маржи, владелец не увидит прямой выгоды).
- **Technical card quality:** 3/10 (Текущие рецепты — это просто текст и картинка, нет связки с ингредиентами и ценами).
- **Costing logic:** 1/10 (Отсутствует).
- **Commercial readiness:** 3/10 (Нужен MVP с фокусом на ТТК и фудкост).
- **MVP focus:** 5/10 (Слишком много фокуса на чек-листы и чаты, слишком мало на рецептуры и деньги).

## 3. Что сейчас сделано хорошо
- **Фокус на смену (Shift):** Идея с "Shift command screen" отличная. Понимание, кто на смене, стоп-листы и задачи.
- **Мобильный подход (Mobile-first):** PWA и нижняя навигация идеально подходят для работы на кухне одной рукой.
- **Сигналы склада:** Быстрая отправка сигнала "заканчивается продукт" без необходимости делать полную инвентаризацию.
- **Инструкции по цехам:** Разделение чек-листов на зоны (Холодный, Горячий, Суши).
- **Быстрые действия (Quick Actions):** Кнопка FAB для быстрых сигналов — топ.

## 4. Главные слабости сейчас
- **Монолитный код:** `src/main.jsx` содержит всю логику, UI и состояние. Это убьет проект при добавлении новых фич.
- **Отсутствие Data Model для ТТК:** Рецепты сейчас хардкодятся как плоские объекты без связи с таблицей ингредиентов.
- **Нет калькулятора Food Cost:** Владельцы платят за экономию денег. Без учета себестоимости ценность продукта резко падает.
- **Нет CRUD-интерфейсов:** Нельзя создать рецепт, добавить ингредиент, изменить цену поставщика прямо из UI (всё в заглушках).
- **Перегрузка стейтов:** Использование 20+ `useState` в главном компоненте делает работу медленной и сложной для рефакторинга.

## 5. Критически недостающие функции (Critical missing features)
1. **Редактор ТТК (Технико-технологических карт):** С возможностью добавлять ингредиенты, их брутто/нетто, и выход в граммах.
2. **База ингредиентов с ценами:** Связка "Продукт -> Поставщик -> Цена за кг/шт".
3. **Калькулятор Food Cost (Себестоимости) и Маржи:** Автоматический пересчет стоимости порции при изменении цен поставщиков.
4. **Управление меню:** Назначение цены продажи и подсчет наценки.
5. **Ролевая модель в UI:** Сейчас интерфейс один для всех. Повару не нужны настройки ресторана, а шефу нужен полный контроль.

## 6. Screen-by-screen audit
### Экран "Смена" (Shift)
- **Оценка:** Хорошо.
- **Проблема:** Может быть перегружен, если задач много.
- **Решение:** Добавить сворачивающиеся блоки (accordion) для задач с низким приоритетом.
### Экран "ТТК" (Recipes)
- **Оценка:** Очень слабо.
- **Проблема:** Это просто список карточек с текстом. Нет состава, нет расчета стоимости.
- **Решение:** Полностью переработать. Добавить вкладки: "Состав", "Приготовление", "Себестоимость".
### Экран "Склад" (Inventory)
- **Оценка:** Нормально для сигналов.
- **Проблема:** Нет возможности управлять приходами и остатками.
- **Решение:** Добавить функционал внесения инвентаризации (цифрами, а не только статусами "мало/норма").
### Экран "Цеха" (Stations)
- **Оценка:** Хорошо.
- **Проблема:** Инструкции статичны.
- **Решение:** Дать возможность шефу редактировать чек-листы прямо с телефона.
### Экран "Чат" (Chat)
- **Оценка:** Лишнее на данном этапе.
- **Решение:** Скрыть (Remove/Hide). Повара используют Telegram/WhatsApp. Не тратьте время на чат в MVP, сфокусируйтесь на ТТК.

## 7. Идеальные пользовательские пути (Ideal user flows)
- **Create recipe:** Кнопка "Создать" -> Ввод названия и фото -> Выбор категории -> Добавление ингредиентов из базы (с авто-подтягиванием цены) -> Ввод брутто/нетто -> Описание шагов -> Назначение цены продажи -> Сохранение.
- **Add ingredients:** Переход в "Склад/База" -> "Новый продукт" -> Название, ед. измерения, коэффициент потерь, поставщик, цена.
- **Calculate cost:** Происходит автоматически при сохранении ТТК или обновлении цены ингредиента. Система подсвечивает красным рецепты, где Food Cost превысил норму (например, >30%).
- **Generate technical card:** Кнопка "Экспорт в PDF" в рецепте -> Формируется строгий A4 документ с граммовками для распечатки на кухню (без цен для обычных поваров, с ценами для шефа).

## 8. Рекомендуемый MVP (Simplest sellable version)
**Core user:** Шеф-повар или владелец небольшого ресторана/кафе (1-3 точки).
**Core pain:** Бардак в рецептурах, ТТК в Excel, непонятная себестоимость блюд, потери времени на обучение новых поваров.
**Core promise:** "Все рецепты, себестоимость и чек-листы вашей кухни в одном удобном приложении в кармане".
**5 must-have features:**
1. Интерактивная база ТТК (рецепты + фото + шаги).
2. Калькулятор себестоимости и маржи (Food Cost).
3. База ингредиентов с ценами поставщиков.
4. Экспорт ТТК в PDF (для печати).
5. Чек-листы открытия/закрытия смены.
**5 features to avoid for now:**
1. Встроенный мессенджер (Чат).
2. Табельный учет рабочего времени (Сложно, есть другие тулзы).
3. Интеграция с кассой (POS) (Слишком рано).
4. Сложный складской учет FIFO (Оставьте только простые сигналы и инвентаризацию).
5. Графики смен.

## 9. Product roadmap (Дорожная карта)
- **Phase 1 (Fix now):** Рефакторинг `src/main.jsx`. Разделение на компоненты, внедрение стейт-менеджера (Zustand). Подключение реального CRUD к Supabase.
- **Phase 2 (MVP):** Разработка модуля ТТК и ингредиентов. Калькуляция себестоимости. Авторизация и роли (Шеф / Повар).
- **Phase 3 (Paid beta):** Экспорт в PDF. Сигналы склада. Поиск первых 5-10 ресторанов для тестирования.
- **Phase 4 (Full commercial version):** Подписки (Stripe), мульти-рестораны, продвинутая аналитика изменения цен поставщиков.

## 10. Technical recommendations
- **Frontend:** Разбить `src/main.jsx` на директории (`components/`, `pages/`, `hooks/`, `store/`).
- **State Management:** Заменить множество `useState` на Zustand для глобального состояния (текущая смена, профиль пользователя).
- **Data Model (Supabase):**
  - `ingredients` (id, name, unit, cost_per_unit, supplier_id)
  - `recipes` (id, name, category, prep_time, target_margin)
  - `recipe_ingredients` (recipe_id, ingredient_id, quantity_gross, quantity_net)
- **Auth:** Использовать RLS (Row Level Security) в Supabase с привязкой к `tenant_id` (restaurant_id), чтобы данные разных ресторанов не пересекались.

## 11. UI/UX recommendations
- **Навигация:** Убрать "Чат", добавить "База" (Ингредиенты/Поставщики).
- **Формы:** На мобилках формы добавления ингредиента должны использовать `type="number"` для клавиатуры с цифрами при вводе граммовок.
- **Темная тема:** Обязательно добавить Dark Mode. На кухне часто яркий свет, контрастность важна, но иногда шефы работают в тусклом свете (склады).
- **Empty states:** Если нет рецептов — показать красивую заглушку с кнопкой "Создать первый рецепт" и 1 демо-рецептом.

## 12. Monetization and pricing
- **Free trial:** 14 дней полного доступа (без привязки карты).
- **Tier 1 (Starter):** $29/мес (1 ресторан, до 5 пользователей, базовые ТТК и чек-листы).
- **Tier 2 (Pro):** $79/мес (1 ресторан, безлимит пользователей, продвинутый Food Cost, экспорт PDF, инвентаризация).
- **Tier 3 (Group):** $199/мес (Мульти-локации, централизованная база рецептов).

## 13. Go-to-market suggestions
1. Идти к знакомым шеф-поварам, дать им пользоваться бесплатно в обмен на feedback.
2. Сделать демо-ресторан в вебе, куда можно зайти без регистрации (Try without login) и покликать.
3. Продавать через боль: "Сколько вы теряете на том, что повар кладет на 20 грамм лосося больше из-за того, что нет распечатанной ТТК с фото?".

## 14. Final verdict
- **Стоит ли продолжать?** Однозначно ДА. Рынок ресторанного софта переполнен корпоративным хламом (iiko, r-keeper), который ненавидят повара. Мобильный Chef-first подход — это отличная ниша.
- **Что исправить в первую очередь?** Технический долг (распилить `main.jsx`) и архитектуру базы данных для ТТК (связь Рецепт -> Ингредиент -> Цена).
- **Действие на завтра:** Удалить вкладку Чат. Начать проектировать таблицы ингредиентов и ТТК в Supabase. Создать экран "Калькулятор блюда".
